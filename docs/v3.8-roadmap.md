## 2. Attestation JSON Schema Design

### Design Principles

- Based on A2A Operator Attestation Registry (OATR) concepts extended with our security-test-specific fields.
- JSON Schema draft-2020-12 for maximum tooling compatibility.
- Every test result becomes an `attestation_entry` inside a top-level `attestation_report`.
- Fields are a superset of the current `generate_report()` output, so existing reports can be migrated.

### Schema Location

`schemas/attestation-report.json`

### Top-Level Structure

```
attestation_report
  ├── schema_version        (semver, "1.0.0")
  ├── harness_version       (e.g. "3.8.0")
  ├── suite                 (harness name)
  ├── timestamp             (ISO 8601)
  ├── target                (URL/command tested)
  ├── summary               (total, passed, failed)
  └── entries[]
        ├── test_id         (e.g. "MCP-001")
        ├── category        (e.g. "tool_poisoning")
        ├── result          (pass | fail | error | skip)
        ├── severity        (P0-Critical | P1-High | P2-Medium | P3-Low | P4-Info)
        ├── scope
        │     ├── protocol       (mcp | a2a | l402 | x402 | ...)
        │     ├── layer          (protocol | operational | decision)
        │     ├── attack_type    (free text, e.g. "tool injection")
        │     └── target_component (e.g. "tools/call endpoint")
        ├── remediation
        │     ├── description    (human-readable fix)
        │     ├── references[]   (URLs to specs, CVEs, etc.)
        │     └── priority       (immediate | next-release | backlog)
        ├── timestamp       (per-entry ISO 8601)
        ├── elapsed_s       (float)
        ├── agent_identity
        │     ├── agent_card_url (optional, A2A agent card)
        │     ├── operator_id    (optional, OATR operator)
        │     └── trust_score    (optional, 0-100)
        ├── protocol_version (wire protocol version tested)
        ├── owasp_asi       (OWASP ASI mapping, e.g. "ASI04")
        ├── statistical     (optional)
        │     ├── trials
        │     ├── pass_rate
        │     └── confidence_interval
        └── details         (free text or structured evidence)
```

### Compatibility with Current Reports

The existing `generate_report()` in each harness produces:
```json
{ "suite", "timestamp", "summary", "results": [{ "test_id", "name", "category", "owasp_asi", "severity", "passed", "details", ... }] }
```

The attestation schema is a strict superset. A migration function will:
1. Wrap existing results into `entries[]` with new fields defaulted.
2. Map `passed: true/false` to `result: "pass"/"fail"`.
3. Add `scope` and `remediation` from a lookup table keyed on `test_id`.

## 3. Scope & Localization Annotations

Each test entry gains two structured sub-objects:

### `scope` - Where the attack targets
- `protocol`: Which wire protocol (mcp, a2a, l402, x402, platform, framework)
- `layer`: Which of the three security layers (protocol, operational, decision)
- `attack_type`: Human-readable attack classification
- `target_component`: Specific endpoint/method/feature targeted

### `remediation` - What to do about it
- `description`: Actionable fix in plain language
- `references[]`: Links to relevant specs, CVEs, OWASP entries
- `priority`: Triage hint (immediate / next-release / backlog)

These annotations make reports self-contained: a security engineer can read a single entry and understand what happened, why it matters, and what to do.

## 4. MoltBridge + TrustAgentAI Mapping

### Overview

Both MoltBridge and TrustAgentAI need to consume attestation output to:
- **MoltBridge**: Aggregate security posture across multiple agents in a deployment, feed into compliance dashboards.
- **TrustAgentAI**: Map attestation entries to trust scores, update agent trust profiles, trigger remediation workflows.

### Integration Points

| Consumer | Input | Mapping | Output |
|----------|-------|---------|--------|
| MoltBridge | `attestation_report` JSON | Each `entry` maps to a posture finding; `scope.layer` maps to MoltBridge risk categories | Aggregated risk score per agent/deployment |
| TrustAgentAI | `attestation_report` JSON | `agent_identity` links to TrustAgentAI agent profiles; `result` + `severity` feed trust scoring | Updated trust profile, remediation tickets |

### Mapping Rules

1. **Agent identity resolution**: `agent_identity.agent_card_url` is the primary key. If absent, fall back to `target` URL.
2. **Severity to risk weight**: P0=1.0, P1=0.75, P2=0.5, P3=0.25, P4=0.0
3. **Layer aggregation**: Each layer (protocol, operational, decision) gets an independent sub-score.
4. **Pass rate**: When `statistical` data is present, use `pass_rate` instead of binary pass/fail for scoring.

### Consumption Example

```python
import json

with open("attestation-report.json") as f:
    report = json.load(f)

for entry in report["entries"]:
    agent = entry.get("agent_identity", {}).get("agent_card_url", report["target"])
    layer = entry["scope"]["layer"]
    severity_weight = {"P0-Critical": 1.0, "P1-High": 0.75, "P2-Medium": 0.5, "P3-Low": 0.25, "P4-Info": 0.0}
    weight = severity_weight.get(entry["severity"], 0.5)
    
    if entry["result"] == "fail":
        # Deduct from trust score
        deduction = weight * 10  # scale factor
        update_trust_score(agent, layer, -deduction)
        
    if entry["remediation"]["priority"] == "immediate":
        create_remediation_ticket(agent, entry)
```

### Delivery Format

- **File**: JSON file conforming to `schemas/attestation-report.json`
- **Streaming**: One JSON object per line (JSONL) for real-time consumers
- **API**: Future v3.9+ will add webhook delivery
